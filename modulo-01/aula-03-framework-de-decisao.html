Por que precisa de framework

Os dois extremos que matam o design

  • Sempre nível 1: você se contenta com o que dá pra fazer no Figma.
  • Sempre nível 4: você overengeneera o que era pra ser um esboço.
  • Framework = critério pra escolher caso a caso.
O framework

As 5 perguntas

  • 1. O que estou validando?
  • 2. Quem vai usar o protótipo?
  • 3. O dado real muda o desenho?
  • 4. O erro custa caro?
  • 5. Quanto tempo eu tenho?
Pergunta 1

O que estou validando?

  • Visual (hierarquia, contraste, marca) → Nível 1
  • Fluxo (caminho, onboarding, navegação) → Nível 2
  • Comportamento (interação, motion, feedback) → Nível 3
  • Experiência (dado real, ponta a ponta, edge case) → Nível 4
Pergunta 2

Quem vai usar o protótipo?

  • Você mesma (pra pensar): 1–2
  • Time de produto (pra alinhar): 2–3
  • Usuário em teste (pra validar): 3–4
  • Stakeholder cético (pra convencer): 4 — precisa ver rodar
Pergunta 3

O dado real muda o desenho?

● NÃO

Nível 1–2 basta

Telas de marketing, dashboard estático, fluxo de cadastro simples. O lorem ipsum não esconde nada.

● SIM

Nível 3–4 obrigatório

Listas, filtros, busca, conteúdo gerado por IA, feed, qualquer coisa que muda com o que o usuário fez antes. Sem dado real, você está projetando uma fantasia.

Pergunta 4

O erro custa caro?

  • Erro visual = revert em 10 minutos → 1–2 basta
  • Erro de fluxo = retrabalho em sprint → suba pra 3–4
  • Erro em produção = incidente, suporte, reputação → 4 obrigatório

Aplicando

3 casos práticos

  • Redesign de empty state → visual, sem dado real → Nível 1
  • Novo fluxo de onboarding → comportamento, usuário externo → Nível 3 ou 4
  • Feature de IA → dado real é central, sem ele não dá pra projetar → Nível 4 sempre
